## Problem Statement

Three pain points have shown up while running the existing `implement` orchestrator skill:

1. **Questioning is unpredictable.** Some skills (e.g. `grill-me`, `slice`) interview the user with the multi-choice `AskUserQuestion` UI; others don't. The user prefers free-form text answers — multi-choice is faster to click but loses fidelity. There's no shared rule across skills, so the behaviour reads as random.

2. **The orchestrator's context bloats.** `implement` only delegates to sub-agents when *multiple* slices are unblocked at once (step 4a). Single unblocked slices get executed inline (step 4). For features whose DAG unblocks one slice at a time, the orchestrator does *all* the implementation work itself, accumulating code, tests, and exploration in its own context.

3. **Human review and QA route through the bloated orchestrator.** When a slice has `Human checkpoint: yes` or end-of-run QA happens, control sits in the orchestrator's session. Any user feedback ("change X") risks being handled inline, growing the orchestrator's context further with code it should never have read.

## Solution

Two changes:

**A. Add a hard "free-form questions only" rule to interview-heavy skills** (`grill-me` and `slice`). They explicitly forbid use of the `AskUserQuestion` multi-choice UI and require plain-text questioning. Other skills are unaffected.

**B. Build a new `dispatch` skill** that coexists with `implement` (the user kept `implement` for now; the two will diverge). `dispatch` consumes the same `docs/{feature}/{feature}.tasks.md` DAG produced by `slice`, but its orchestrator stays structurally lightweight: it never executes slice work, only dispatches sub-agents, parses their structured replies, and updates bookkeeping files.

## User Stories

1. As a developer using `/grill-me`, I want the assistant to ask questions in plain text, so that I can give nuanced free-form answers without losing fidelity to multi-choice picks.
2. As a developer using `/slice`, I want step-5 grilling to use plain text only, for the same reason.
3. As a developer running a feature DAG, I want the orchestrator's context to stay small regardless of how many slices unblock sequentially, so that long features don't degrade into truncated, confused responses near the end.
4. As a developer running parallel slices, I want each slice to execute in an isolated git worktree, so that two slices touching adjacent code never produce broken intermediate state.
5. As a developer reviewing a `Human checkpoint: yes` slice, I want the orchestrator to surface only the agent's `needs-review` reason plus pointers to the log and changed files, so that the orchestrator never reads the slice's code.
6. As a developer requesting changes after review, I want my feedback dispatched to a fresh sub-agent rather than executed inline, so that orchestrator context stays bounded.
7. As a developer running `dispatch`, I want Opus on the orchestrator and Sonnet on the implementers by default, so that judgment work and volume work each run on the right-sized model.
8. As a developer with parallel slices that finish on different worktree branches, I want clean merges to happen automatically and conflicts to spawn a merge-fix sub-agent, so that the autonomous flow continues without burdening the orchestrator with conflict resolution.
9. As a developer at end of run, I want a QA plan generated by the (still-lightweight) orchestrator, so that I can take it into a fresh session for verification.
10. As a developer who wants Codex-backed implementers in future, I want `dispatch`'s prose to remain backend-agnostic, so that adding Codex later is an additive change to the SKILL.md rather than a rewrite.

## Implementation Decisions

### Module: `grill-me/SKILL.md` (edit)

Add a single rule near the top of the body: questions to the user must be free-form text in the chat. Do not use the `AskUserQuestion` multi-choice UI under any circumstance.

### Module: `slice/SKILL.md` (edit)

Add the same free-form rule, scoped to step 5 (the grilling questions) but worded as a hard constraint applying to the whole skill. Update step 7's "stop" message to mention `/dispatch` as an alternative next-step option alongside `/implement`.

### Module: `dispatch/SKILL.md` (new)

The new orchestrator skill. Key shape:

- **Roles:** orchestrator dispatches; slice agents execute. Orchestrator never writes implementation code, never reads slice code (except via the agent's `<status>` reply or log entries it explicitly opens to compose the QA plan).
- **Always delegate.** Even a single unblocked slice goes to a sub-agent. There is no inline-execution branch.
- **Minimal handoff.** The orchestrator's spawn message contains only: the slice slug, the absolute path to the tasks file, the absolute path to the log file, and a pointer to `slice-prompt.md`. The sub-agent reads the tasks file itself and locates its slice by slug.
- **Worktree isolation.** Parallel sub-agent spawns use `isolation: "worktree"` on the `Agent` tool. Single spawns may run in-place.
- **Structured handoff via tags.** Sub-agents emit one of: `<status>done</status>`, `<status reason="...">needs-review</status>`, or `<status reason="...">blocked</status>`. The orchestrator regex-parses the tag — nothing else.
- **Human checkpoints.** On `needs-review`, the orchestrator surfaces only the `reason` attribute plus the path to the relevant log entry and the list of changed files (which it can pull from `git diff --name-only` against the worktree branch — no code reads). Approval marks the slice `done`. Change requests dispatch a new sub-agent with the user's feedback inlined into the brief.
- **Merging.** When a worktree-isolated sub-agent finishes `done`, the orchestrator runs `git merge --no-ff <branch>`. Clean merges proceed silently. Conflicts spawn a dedicated merge-fix sub-agent, briefed with the conflict markers, the two log entries, and the slice specs of both sides; that sub-agent emits the same `<status>` tags and is treated like any other slice agent (clean → merged; needs-review → surface to user).
- **QA plan generation.** Stays in the orchestrator. It reads the tasks file and log file once, writes `{feature}.qa.md`, then stops completely.
- **Default models.** Orchestrator runs on Opus; sub-agents spawn with `model: "sonnet"`. Both are overridable.
- **Backend.** Pure Claude via the `Agent` tool. The SKILL.md describes dispatch in backend-agnostic prose so a future Codex backend can be added by editing the dispatch tactic, not the structure.

### Module: `dispatch/resources/slice-prompt.md` (new)

The role definition the orchestrator points sub-agents at. Defines:

- The agent's job (find the slice in the tasks file by slug, implement it with TDD, write a log entry, update its row's status).
- Required reads: the tasks file (find your slice), the log file (read prior entries before starting), `tdd-loop.md`, and `log-format.md`.
- Output contract: emit exactly one `<status>` tag at the end. Reason attribute required for `needs-review` and `blocked`.
- Constraint: update only this slice's row in the tasks file; never touch siblings.

## Testing Decisions

These skills are markdown prose, not code — there is no test suite to extend. Validation is empirical:

- **Free-form rule:** trigger `/grill-me` and `/slice` step 5 in a fresh session; confirm questions arrive as plain text and the multi-choice UI never renders.
- **`dispatch` end-to-end:** run it on a small 2–3 slice DAG with one parallel pair and one `Human checkpoint: yes` slice. Confirm: (a) orchestrator's context window stays roughly flat across slices (inspect via `/usage` or context indicator), (b) parallel slices land in separate worktrees, (c) the human-checkpoint pause surfaces only the reason + pointers (no inlined code), (d) end-of-run QA plan is written and the orchestrator stops.
- **Merge conflict path:** construct a DAG where two parallel slices deliberately touch the same line, run `dispatch`, confirm the merge-fix sub-agent spawns and resolves (or escalates).
- **Status-tag parsing:** unit-level — feed the orchestrator's parser sample stdout containing each tag variant and confirm it picks the right branch.

The course-video-manager `.sandcastle/` setup is the closest prior art for the multi-agent loop pattern; reference it when ambiguity arises about prompt-file shape or tag conventions.

## Out of Scope

- **Codex backend.** Deferred until Claude-only `dispatch` has been used long enough to confirm the abstraction is right (Q on Codex parity → "defer").
- **Replacing `implement`.** Coexistence chosen for now; replacement is a possible future cleanup, not part of this work.
- **Autonomous (headless) mode.** Sandcastle's TS-orchestrator pattern is interesting but requires rebuilding human checkpoints; not in scope.
- **A dedicated merger sub-agent for clean merges.** Clean merges are mechanical; only conflicts spawn an agent.
- **Restructuring the existing resources.** `tdd-loop.md`, `log-format.md`, `qa-template.md` are duplicated rather than refactored into a shared location — the skills are expected to diverge.
- **Updating `spec/SKILL.md`.** It does not interview and does not orchestrate; nothing changes.
